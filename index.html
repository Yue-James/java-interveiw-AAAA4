FIT1051 终极面试准备指南

Ultimate Interview Prep Guide (Student's Version)

这不是考试，这是一场关于你代码的对话。这份指南是你的思路整理工具，帮助你自信地应对任何问题。

板块一：程序设计宏观讲解

Area 1: Explaining Your Program Designs

The interviewer wants to know: Can you clearly describe your project's structure from a high level? This proves you conceived the overall design.
面试官想知道的：你能从“上帝视角”清晰地描述你的项目结构吗？这能证明项目的整体设计是你构思的。

English Talking Points

  • The Big Picture: "My project is a 7-day student life simulator. The core idea was to separate the different parts of the game to keep the code clean. I have four main components that work together."
  • The Components & Their Jobs:
    • "GenZLifeSimulator.java is just the entry point. Its main method kicks everything off, and that's it."
    • "Student.java is where all the data lives. It's like the character sheet. All the stats—energy, money, etc.—are private fields, which is an example of encapsulation. I did this so I could control how stats are changed. For example, in the addEnergy method, I use Math.max(0, ...) to make sure energy can never accidentally drop below zero."
    • "Event.java and ActivityEvents.java define all the possible actions. Event is an abstract class that acts as a blueprint for any activity."
    • "Simulator.java is the engine. It runs the main game loop, shows menus, gets user input, and applies the game's rules."

中文谈话要点

  • 宏观介绍：「我的项目是一个7天的学生生活模拟器。核心思路是把游戏的不同部分分开，让代码保持整洁。我主要有四个部分协同工作。」
  • 组件和分工：
    • 「GenZLifeSimulator.java 只是程序的入口，main方法就在这里，它负责启动游戏，仅此而已。」
    • 「Student.java是所有数据的存放地，就像角色的属性面板。所有的属性——精力、金钱等——都是私有变量，这是一个封装 (Encapsulation) 的例子。我这样做是为了能控制属性如何被修改。比如，在addEnergy方法里，我用了Math.max(0, ...)来确保精力值永远不会意外地变成负数。」
    • 「Event.java和ActivityEvents.java定义了所有玩家能做的动作。Event是一个abstract(抽象)父类，作为一个活动蓝图。」
    • 「Simulator.java是游戏的引擎。它负责运行主游戏循环、显示菜单、获取用户输入以及应用游戏规则。」

板块二：编码决策深度剖析

Area 2: Discussing Your Coding Decisions

The interviewer wants to know: Is there a reason behind each choice you made? This demonstrates your depth of thought.
面试官想知道的：你做出的每一个选择背后都有理由吗？这能体现你的思考深度。

English Talking Points

  • Why an abstract class for Event, not an interface? "I specifically chose an abstract class because all events share some *implemented code*. The apply method in Event.java contains the logic for the random mishap check. This code is the same for every single event. An interface is just a contract with no implementation, so if I used an interface, I'd have to copy and paste that mishap logic into every event subclass. The abstract class lets me write that shared code just once."
  • Why use final on the Event.apply() method? "I made the apply method final on purpose. It forces every event to follow the exact same sequence: first apply base effects, then check for a mishap. I didn't want subclasses to mess up this order or accidentally break the core game rule, so final protects that structure."
  • Why Array vs. ArrayList? "I used a standard Array for stat history in the Student class because the game is always 7 days long, so the size is fixed and known. It's simple and direct. I used an ArrayList for the daily activities in the Simulator because the player might choose one or two activities, so the size is variable. An ArrayList is flexible and handles that easily."

中文谈话要点

  • 为什么`Event`用抽象类，而不是接口？「我特意选了抽象类，因为所有事件都共享一段*已经实现了的代码*。`Event.java`里的`apply`方法包含了检查随机意外的逻辑。这段代码对每个事件都完全一样。接口只是一个没有具体实现的契约，所以如果我用了接口，我就必须把那段意外逻辑复制粘贴到每一个事件子类里。用抽象类让我可以把这段共享代码只写一次。」
  • 为什么`Event.apply()`方法要用`final`关键字？「我是故意把`apply`方法设为`final`的。它强制每个事件都遵循完全相同的执行顺序：先应用基础效果，再检查意外事件。我不希望任何子类搞乱这个顺序，或者意外地破坏游戏的核心规则，所以`final`起到了保护这个结构的作用。」
  • 为什么混用`Array`和`ArrayList`？「我在`Student`类里用普通`Array`来记录历史数据，因为游戏永远是7天，长度是固定的，用数组简单直接。我在`Simulator`里用`ArrayList`来装每天的活动，因为玩家可能选一个或两个活动，数量是可变的，`ArrayList`的灵活性正好能处理这种情况。」

板块三：具体代码逻辑解读

Area 3: Explaining Your Code

The interviewer wants to know: Can you, as the author, explain the logic of any part of your code?
面试官想知道的：你能像作者一样，解释你代码中任意一段的逻辑吗？

English Talking Points

  • Explaining Simulator.getValidatedInput(): "This is a helper method I wrote to make sure the user can't crash the program or enter invalid choices. It's an infinite while (true) loop that only breaks when valid input is received. I wrapped the Integer.parseInt call in a try-catch block. If the user types text instead of a number, it catches the NumberFormatException and prints an error instead of crashing. Then, I check if the number is within the min and max range I provided. Only when a valid integer in the correct range is entered does the method return the value and break the loop."
  • Explaining the Perk System in Simulator.performActivity(): "After an event's apply method is called, I have this line: if (chosenPerk != null) { chosenPerk.applyBonus(...) }. This is where polymorphism comes in. The chosenPerk variable is of type Perk (the interface), but the object it holds could be an AcademicGeniusPerk or a HardWorkerPerk. When I call applyBonus, Java automatically runs the version of the method from the *actual object* that was created at the start of the game. So this one line of code works for all four different perks without needing any if-else checks."

中文谈话要点

  • 解读 `Simulator.getValidatedInput()`: 「这是我写的一个辅助方法，为了确保用户不能通过乱输来让程序崩溃。它是一个无限`while (true)`循环，只有在用户输入了有效内容后才会跳出。我把`Integer.parseInt`放在一个`try-catch`代码块里。如果用户输入了字母而不是数字，它就会捕获`NumberFormatException`异常并提示错误，而不是直接崩溃。接着，我还会检查这个数字是否在我设定的`min`和`max`范围内。只有当一个在正确范围内的有效数字被输入后，这个方法才会`return`数值并结束循环。」
  • 解读`Perk`系统在`Simulator.performActivity()`中的体现：「在一个事件的`apply`方法被调用后，我有这样一行代码：if (chosenPerk != null) { chosenPerk.applyBonus(...) }。这正是多态 (Polymorphism) 的体现。chosenPerk这个变量的类型是Perk(接口)，但它实际指向的对象可能是AcademicGeniusPerk或者HardWorkerPerk。当我调用applyBonus时，Java会自动执行游戏开始时创建的*那个具体对象*的方法。所以，这一行代码就能处理所有四种不同的专长，完全不需要任何if-else判断。」

板块四：核心编程概念关联

Area 4: Explaining Coding Concepts

The interviewer wants to know: Can you connect the theoretical concepts from the course (like inheritance, polymorphism) with your own practical code?
面试官想知道的：你能把课程中学到的理论知识（如继承、多态）和你自己的代码实践联系起来吗？

English Talking Points

  • Inheritance: "I used inheritance for my Event structure. For example, ActivityEvents.AllNighterEvent extends Event. This means it automatically gets the eventName field and the apply method from the parent, but it *must* provide its own implementation for the abstract methods like applyBaseEffects."
  • Polymorphism: "The best example is the ArrayList<Event> in the Simulator. I can add a new AttendClassEvent() and a new PartTimeShiftEvent() to the same list. When I loop through it and call activity.apply(), polymorphism ensures the correct version of the code runs for each different object."
  • Abstract Classes vs. Interfaces: "I used both. Event is an abstract class because I wanted to share implemented code (the mishap check logic). Perk is an interface because the different perks don't share any code; they only share a capability or a contract—the promise that they will have an applyBonus method."
  • Enums: "I used enums for Difficulty and EventType. This is much better than using strings or integers because it's type-safe. The compiler ensures I can only use the predefined values, which prevents typos and makes the code much more readable."

中文谈话要点

  • 继承 (Inheritance): 「我用继承来构建我的`Event`体系。比如，`ActivityEvents.AllNighterEvent` `extends` `Event`。这意味着它自动就拥有了父类的`eventName`字段和`apply`方法，但它*必须*为像`applyBaseEffects`这样的抽象方法提供自己的实现。」
  • 多态 (Polymorphism): 「最好的例子就是`Simulator`里的那个`ArrayList<Event>`。我可以把一个`new AttendClassEvent()`和一个`new PartTimeShiftEvent()`对象放进同一个列表里。当我遍历这个列表并调用`activity.apply()`时，多态确保了不同对象的正确代码被执行。」
  • 抽象类 vs. 接口: 「我两个都用到了。`Event`是一个抽象类，因为我想在父类里共享已经实现了的代码（意外检查逻辑）。`Perk`是一个接口，因为不同的专长不共享任何代码，它们只共享一个‘能力’或者说一个‘契约’——就是它们都承诺自己会有一个`applyBonus`方法。」
  • 枚举 (Enums): 「我用枚举来定义`Difficulty`(难度)和`EventType`(事件类型)。这比直接用字符串或者数字要好得多，因为它是类型安全的。编译器会保证我只能使用预设好的那几个值，这能防止我打错字，也让代码的可读性更强。」

板块五：现场编码与修改

Area 5: Modifying Your Code

The interviewer wants to know: Do you truly "own" this code? Can you quickly and accurately modify it under guidance?
面试官想知道的：你是否真的“拥有”这段代码？你能在指导下快速、准确地修改它吗？

English Talking Points

  • Simple Change (e.g., "Change crash energy recovery to 15"): "Okay. That value is a constant at the top of my Simulator.java file. I'll find the line private static final int CRASH_ENERGY_RECOVERY = 10; and change the 10 to a 15."
  • Complex Change (e.g., "Add a rule: if you skip two days in a row, you lose an extra 10 happiness"): "That's an interesting idea. My current daysSkipped variable just counts the total. To check for *consecutive* skips, I'd need to check the history. I could add a new method in the Student class, similar to how hasConsecutiveLowHappiness works, that checks a history of skipped days. So in the Simulator, when the player skips, I would call that new method on the student to see if yesterday was also a skip, and if so, apply the penalty."

中文谈话要点

  • 简单修改 (例如：「把崩溃后恢复的精力改成15」):「好的。那个数值是我在`Simulator.java`文件顶部定义的一个常量。我只要找到`private static final int CRASH_ENERGY_RECOVERY = 10;`这一行，把10改成15就行了。」
  • 复杂修改 (例如：「增加一个规则：如果连续两天‘跳过’，额外扣10点幸福度」):「这是个有趣的想法。我目前的`daysSkipped`变量只统计了总次数。要检查*连续*跳过，我需要去查历史记录。我可以在`Student`类里加一个新方法，就类似于我现在`hasConsecutiveLowHappiness`的实现方式，让它去检查跳过的历史。这样在`Simulator`里，当玩家选择跳过时，我就调用学生对象的这个新方法，看看昨天是不是也跳过了，如果是，就施加惩罚。」

板块六：代码讨论：反思与展望

Area 6: Discussing Your Code

The interviewer wants to know: Can you think critically about your own work? Do you recognize its limitations and have ideas for improvement?
面试官想知道的：你能对自己的工作进行批判性思考吗？你是否认识到代码的局限性并有改进的想法？

English Talking Points

  • The Bug You Fixed: "I'm actually glad you asked. During testing, I found a significant bug in the processCrash logic where a player could get stuck in an infinite 'crash loop' if their happiness hit zero. It was a good lesson in testing edge cases, not just the 'happy path'. I added a condition to give back a little happiness if it was already critically low, which solved the problem."
  • Known Limitations: "A clear limitation is the big switch statement in Simulator that creates the Event objects. It works fine for the current number of events, but it's not very scalable. If this were a bigger project, I'd refactor that using a design pattern like a Factory or a HashMap to make adding new events cleaner."
  • If You Had More Time: "The first thing I'd add is a save/load feature. It would be a great challenge to figure out how to serialize the Student object's state to a file and read it back. Another idea would be to make the events more dynamic, for example, certain events might only become available if you meet a specific stat requirement."

中文谈话要点

  • 你修复的Bug： 「我其实很高兴你问这个。在测试中，我确实在`processCrash`逻辑里发现了一个挺严重的bug，玩家可能会因为幸福度变成0而陷入无限的‘崩溃循环’。这给我上了一课，让我知道测试边缘情况的重要性，不能只测试正常流程。我加了一个条件，在幸福度已经极低时，返还一点幸福度，从而解决了这个问题。」
  • 代码的局限性： 「一个很明显的局限是`Simulator`里面那个根据用户输入创建`Event`对象的`switch`语句。对于目前数量的事件来说它能工作，但扩展性很差。如果这是个更大的项目，我可能会用像工厂模式或者`HashMap`这样的设计来重构它，让添加新事件变得更优雅。」
  • 如果你有更多时间： 「我第一个想加的功能就是保存/读取系统。学习如何把`Student`对象的状态序列化到一个文件里再读回来，会是一个很好的挑战。另一个想法是让事件变得更动态，比如，某些特殊事件只有在你满足特定属性要求后才会解锁。」

板块七：如何应对未知问题

Area 7: Handling "I Don't Know"

The interviewer wants to know: When you encounter something you don't know, do you give up, make things up, or are you honest and try to solve the problem?
面试官想知道的：当你遇到知识盲区时，你是选择放弃、胡编乱造，还是诚实并尝试解决问题？

English Talking Points

  • "That's a good question. I'm not entirely sure about that specific concept off the top of my head. Could you give me a hint, or should I try to walk through my code to figure it out?"
  • "I haven't encountered that term before in my studies. Could you explain what it means? I'm keen to learn."

中文谈话要点

  • 「这是一个好问题。关于这个具体的概念，我一下子不太确定。您能给我一点提示吗，或者我可以通过看代码来尝试想出答案？」
  • 「这个术语我在之前的学习中没有接触过。您能解释一下它的意思吗？我很想学习一下。」
